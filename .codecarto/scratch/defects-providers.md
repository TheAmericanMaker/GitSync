## Rate-limit / retry behavior summary

The four provider clients are **uniformly retry-naive**: every REST call goes through `httpGet/Post/Patch/Put/Delete` in `lib/api/helper.dart` which sets a 10-second timeout and synthesises a fake `http.Response('Error', 408)` on timeout, with no retry, no exponential backoff, no parsing of `Retry-After`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, or GitHub's `x-ratelimit-*` family. A 429 response from any provider lands in the same `else` branch as a 4xx/5xx and surfaces to the user as an empty list, a `null` detail, or a generic "failed" toast. There is a separate `handleIfNetworkError` in helper.dart (used for git clone/push), but none of the provider REST paths invoke it. The git-protocol retry harness (`scheduleNetworkRetryOp`) uses Workmanager with 30s × 2^retryCount backoff capped at 5 retries, but again is git-only. Net effect: under any provider rate-limit pressure or transient 5xx, the UI silently degrades to empty data without distinguishing "no data" from "rate limited" or "instance down" — the contracts phase should treat this as a load-bearing gap fed by `arch-CF1`.
